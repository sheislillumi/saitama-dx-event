# TESTING.md — 通しテスト手順書

このドキュメントは、`index.html` / `exhibitor/` / `visitor/` / `checkin/` / `admin/` の
一連の流れを手動で通しテストするための手順書です。

基本シナリオ(正常系)は「**来場者申込 → QR発行確認 → QRコード受付 → 管理画面への反映**」を
中心に、出展申込フォームとイベントHPの確認も含めた5画面すべてを対象とします。
後半には通信エラー・カメラが使えない場合などの異常系テストもまとめています。

## 0. テスト前の準備

このテストを始める前に、以下が完了している必要があります。詳細手順は README.md を参照してください。

- [ ] `gas/Code.gs` をApps Scriptエディタに貼り付け、`setupSheets` を実行済み
      (Exhibitors / Visitors / Survey / ErrorLog の4シートが作成されている)
- [ ] スクリプトプロパティ `ADMIN_API_KEY` を設定済み(値を控えておく。管理ダッシュボードの
      ログインパスワードとして使用する)
- [ ] GAS Web Appをデプロイ済みで、ウェブアプリのURLを控えている
- [ ] `assets/js/api-client.js` の `GAS_API_CONFIG.BASE_URL` に、上記デプロイURLが設定されている
- [ ] テスト用に受信できるメールアドレスを1つ用意している(来場者申込フォームでの確認メール受信用)
- [ ] QRコード受付画面(`checkin/`)をテストする端末に、カメラが搭載されている
      (スマートフォン、またはWebカメラ付きPC。無い場合は「4-B. カメラが使えない場合」を実施)
- [ ] ローカルで確認する場合は、リポジトリのルートで簡易サーバーを起動しておく
      (例: `python -m http.server 8000` を実行し `http://localhost:8000/` を開く)。
      GitHub Pagesに公開済みの場合は、そのURLでも同様にテストできる

このテストで新しく登録されるデータ(テスト用の出展者・来場者)はスプレッドシート上に
実データとして残ります。テスト後、不要であれば該当行を削除してください。

## 4. QRコード受付画面(checkin/)

### 4-A. カメラで読み取る場合

| # | 手順 | 期待結果 |
|---|---|---|
| 4-1 | `checkin/index.html` を開く | 受付場所セレクト(展示ゾーン/講演会場)・「カメラを起動」ボタン・手動入力欄が表示される |
| 4-2 | 受付場所を選択し、「カメラを起動」ボタンを押す | ブラウザからカメラ利用許可のダイアログが出るので許可する。カメラ映像が表示され、「枠内にQRコードを写してください。」と案内される |
| 4-3 | 3章で発行したQRコード画像を、カメラに写す(スマートフォンやPC画面に表示したQR画像にカメラを向ける) | 自動的に読み取りが行われ、「〇〇様 受付完了」というメッセージ(緑色)が表示される。受付場所・受付時刻も表示される |
| 4-4 | Googleスプレッドシートの `Visitors` シートを確認する | 該当行の「チェックイン状態」が「済」になり、「チェックイン日時」「受付場所」が記録されている |
| 4-5 | 同じQRコードをもう一度カメラに写す(4秒以上間隔を空ける。連続読み取り防止のため直後は無視される仕様) | 「〇〇様は 受付済みです」というメッセージ(オレンジ色)が表示され、チェックイン日時・受付場所が表示される |

### 4-B. カメラが使えない場合(フォールバック)

| # | 手順 | 期待結果 |
|---|---|---|
| 4-6 | ブラウザの設定でカメラへのアクセスを拒否した状態で「カメラを起動」ボタンを押す | 「カメラを起動できませんでした。カメラへのアクセス許可をご確認いただくか、下の手動入力欄をご利用ください。」と表示され、下部の手動入力欄へ誘導される |
| 4-7 | カメラが搭載されていない端末(またはカメラ利用不可の環境)で「カメラを起動」ボタンを押す | 「このブラウザ・端末ではカメラを利用できません。下の手動入力欄をご利用ください。」と表示される |
| 4-8 | 手動入力欄に、3章で発行されたQRトークン(スプレッドシートの `Visitors` シートの「QRトークン」列からコピーするか、メール本文のQRコードをスマートフォンの標準カメラ/QR読み取りアプリで読み取って文字列を確認する)を入力し、「この内容で受付する」ボタンを押す | カメラ経由と同様に「〇〇様 受付完了」が表示され、シートにも反映される |
| 4-9 | 手動入力欄に、存在しないランダムな文字列(例: `not-a-real-token-123`)を入力して送信する | 「未登録のQRコードです」というエラーメッセージ(赤色)が表示され、「下の「手動で受付する」欄にQRトークンや来場者IDを入力してご確認ください。」と案内される |
| 4-10 | 手動入力欄を空欄のまま「この内容で受付する」ボタンを押す | 「QRトークンまたは来場者IDを入力してください。」と表示され、送信されない |

## 6. 異常系テスト(通信エラー等)

正常系のテストとは別に、以下の異常系についても確認してください。

### 6-1. 通信エラー(GAS APIに到達できない場合)

| # | 手順 | 期待結果 |
|---|---|---|
| 6-1-1 | ブラウザの開発者ツールで「オフライン」モードに切り替える(Chromeの場合: DevTools > Network タブ > Throttling を「Offline」にする)、または `assets/js/api-client.js` の `GAS_API_CONFIG.BASE_URL` を一時的に存在しないURLに書き換える | 出展申込フォーム・来場者申込フォームで送信すると「通信エラーが発生しました。ネットワーク環境をご確認のうえ、再度お試しください。」が表示される(画面が真っ白になったり、何も表示されずに固まったりしない) |
| 6-1-2 | 同様にオフライン状態で、QRコード受付画面から手動入力で送信する | 「通信エラーが発生しました」の旨のメッセージが表示される |
| 6-1-3 | 同様にオフライン状態で、管理ダッシュボードにログインを試みる | 「通信エラーが発生しました。ネットワーク環境をご確認ください。」が表示される |
| 6-1-4 | オンラインに戻し、`GAS_API_CONFIG.BASE_URL` を正しい値に戻して、6-1-1〜6-1-3を再実行する | いずれも正常に動作することを確認する |

| # | 手順 | 期待結果 |

## 7. 付録: GASエディタ上での補助テスト(フロントを介さない検証)

フロントエンド側の問題なのか、GAS側の問題なのかを切り分けたい場合は、
`gas/Code.gs` の末尾にあるテスト関数を直接実行するとよい(Apps Script エディタの
関数選択ドロップダウンから選択して [実行])。

| 関数名 | 用途 |
|---|---|
| `test_registerExhibitor` | 出展申込登録(バックエンド単体)の確認 |
| `test_registerVisitor(testEmail)` | 来場者登録の確認。引数に実在するメールアドレスを渡すと、実際に確認メールの送信まで確認できる(例: `test_registerVisitor('you@example.com')`) |
| `test_checkin` | 来場者登録 → チェックイン → 重複チェックインの一連の流れと、checkinTimeの書式一致を自動判定 |
| `test_getStats` | 集計取得の確認 |
| `test_phoneNumberLeadingZero_` | 電話番号(ハイフンなし)の先頭0が失われないことの回帰テスト |
| `fixPhoneNumberFormat_` | (一時的な修正用)既存データの電話番号列をプレインテキスト形式に直す |

実行結果は Apps Script エディタの [実行数] タブ、またはログ表示(`Ctrl+Enter`)で確認できる。
